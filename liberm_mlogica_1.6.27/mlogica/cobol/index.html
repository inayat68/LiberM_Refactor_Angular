    Working with Veryant isCOBOL

    LIBER*M is a 100% Java framework, therefore it works with any JVM compliant programming language. In the context of a mainframe migration, you may decide not to convert your existing COBOL code to Java but rather retain it using, indeed, a COBOL compiler for the JVM.

    LIBER*M provide a set of features to integrate Veryant isCOBOL.

    !> isCOBOL is a Veryant product: MLOGICA is not responsible for any defect or malfunction caused by the isCOBOL compiler or runtime. For support on the Veryant product, please refer to your veryant license subscription.

    Building COBOL programs

    When compiling cobol programs, it is important to specify as few options to the isCOBOL compiler so to ensure correct cooperation between the generated code and LIBER*M.

    Batch programs

    For batch programs, you need to specify:

    Compiler option Description Mandatory
    -cax Use external file handler yes
    -cv IBM Compatibility flag yes
    -cva IBM arithmetic compatibility yes
    -d Include basic debug information debug
    -dcii Use IBM sign encoding and IBM COMP sizes yes
    -dv=0 Initialize to WS to low-values yes
    -jc Generate the ‘.class’ file yes
    -jj Generate the ‘.java‘ file debug
    -jo=-g Passes '-g' to the java compiler debug
    -m1 Put all of WORKING-STORAGE into a contiguous block of memory. yes
    -smat Allows the mixing of source files and copybooks formats no

    Example:

    Bash
    -c=iscobol.batch.properties -cax -ce=cpy -cv -cva -d -dcii -dv=0 -jc -jj -lo=list -m1 -od=target/classes -smat -sp=src/main/cobol/cpy

    Also, the iscobol.properties passed to the compiler must include:

    Properties
    iscobol.compiler.outdd=SYSOUT

    Online programs

    For batch programs, you need to specify:

    Compiler option Description Mandatory
    -cv IBM Compatibility flag yes
    -cva IBM arithmetic compatibility yes
    -d Include basic debug information debug
    -dcii Use IBM sign encoding and IBM COMP sizes yes
    -dv=0 Initialize to WS to low-values yes
    -jc Generate the ‘.class’ file yes
    -jj Generate the ‘.java‘ file debug
    -jo=-g Passes '-g' to the java compiler debug
    -m1 Put all of WORKING-STORAGE into a contiguous block of memory. yes
    -smat Allows the mixing of source files and copybooks formats no

    Also, the iscobol.properties passed to the compiler must include: Example:

    Bash
    -c=iscobol.cics.properties -ce=cpy -cv -cva -d -dcii -dv=0 -eo=errs -esme=100 -jc -jj -lo=list -m1 -od=target/classes -smat -sp=src/main/cobol/cpy
    Properties
    iscobol.compiler.outdd=SYSOUT
iscobol.compiler.custompreproc=com.nib.supernaut.cobol.precompiler.CicsPrecompiler

    CICS Precompiler

    From online COBOL programs you can invoke LIBER*TP API. To do this you leverage the isCOBOL OO COBOL syntaxes.

    However, this is not necessary, because LIBERTP provides a CICS precompilers that translates EXEC CICS verbs into the corresponding LIBERTP API.

    For the list of CICS command supported by LIBER*TP, see here.

    E/SQL interface

    The E/SQL interface is managed by the isCOBOL compiler, that translates the EXEC SQL statements into corresponding JDBC calls at compile time.

    LIBERM components make sure that the isCOBOL generated JDBC commands are executed over the database connections of the running job or transaction. Therefore, you don't have to care about connecting the database or managing the transaction because it is done by the LIBERM framework.

    Please note that the isCOBOL precompiler do not alter or manipulate the SQL command you provided. This means that the SQL statement must conform the target database syntax. So, if are migrating a COBOL/DB2 application to Java with PostgreSQL, you must convert the DB2 SQL statements into PGSQL compatible ones. The same applies to SQLCODEs, that are not translated by the isCOBOL runtime.

    Online Debug

    1) Start the run region node with the JVM option -Discobol.rundebug=2 2) Connect the terminal and start the first transaction. COBOL runtime should write the message Debugger listening on port 9999 ... in the log. 3) Attach with the desired debugger (Eclipse or isCOBOL), as described below.

    isCOBOL debugger

    Execute the following command in the isCOBOL Evolve Shell to start the debug session:

    Text Only
    iscrun -d -r localhost 9999

    To debug a specific program:

    1) set the breakpoint on the desired program name, thur the menu Breakpoint > Set, select Program and type in the program name. 2) resume the execution with the Run > Continue menu command. 3) use the application normally to reach the desired program: the debugger will stop at the breakpoint.

    Eclipse

    Define a Debug configuration in Eclipse with the following parameters:

    • Type: iSCOBOL remote application
    • Host: localhost
    • Port: 9999

    To debug a specific program:

    1) connect the remote debugger: when the debug starts and the first program is shown in the eclipse debugger, resume execution with Run > Resume 2) navigate the app as desired and just before entering the transaction than triggers the desired program, suspend the debugger with Run > Suspend 3) now enter the necessary PFKEY to start the next transaction and the debugger will stop at the next COBOL program
