    Connectivity

    Server Components

    LIBER*Batch relies on various supporting components for proper functionality, including:

    • Backbone
    • Long Term Job Output Queue
    • Logging
    • Measuring and Monitoring

    Backbone

    The LIBERBatch backbone is a core component that enables node interoperability and clustering within LIBERBatch.

    The backbone is currently available only on Redis. The selected backbone is configured in the LIBER*Batch configuration file by specifying the Redis configuration file.

    Redis Backbone

    LIBER*Batch use a Redis backbone by default(recommended), no setting is required. Just provide the appropriate Redis client configuration file.

    YAML
    motorhead:
  # ...
  redis:
    config: 'file:./redisson-localhost.yaml'```

    LIBER*Batch utilizes the Redisson client (Redisson documentation), requiring configuration in the expected format. Example Redisson configuration:

    YAML
    singleServerConfig:
  address: "redis://127.0.0.1:6379"

    Refer to the Redisson configuration guide for further details.

    Long Term Job Output Queue

    LIBERBatch supports customizable long-term job output queue. By default, LIBERBatch stores the information of the jobs executed in the Redis backbone.

    But it is possible to use a custom long term storage manager, by implementing a manager class that implements the interface com.nib.motorhead.ILongTermStorage and configuring it on the LIBER*Batch configuration file, specifying its class name and the configuration data:

    YAML
    motorhead:
  long-term-storage:
    class: "org.sample.MyLTSManager"
    config: "my config data"

    LIBER*Batch includes by default a Rdbms long term storage that stores job information in relational databases such as PgSql, MySql, DB2, MSSQL and Oracle DB. To activate it, configure as follows:

    YAML
    motorhead:
  long-term-storage:
    class: "com.nib.motorhead.rdbms.RdbmsLongTermStorage"
    config: "jdbc:postgresql://localhost/postgres?user=postgres&password=postgres&ssl=false&currentSchema=darkside"

    Logging

    In a production environment, logs should be shipped to an external aggregator. LIBERBatch leverages SLF4J, allowing integration with any Java logging framework. By default, LIBERBatch uses Logback.

    To configure logging, specify the Logback configuration file in the logging section of the LIBER*Batch configuration file.

    YAML
    logging:
  config: file:./logback.xml
  level:
    com.myapp: INFO

    Example Logback configuration for Loki:

    XML
    <configuration>
    <define name="hostname"
            class="ch.qos.logback.core.property.CanonicalHostNamePropertyDefiner"/>
    <appender name="STDOUT"
              class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <pattern>[%date{ISO8601}][%-24t][%-5level] %msg%n</pattern>
        </encoder>
    </appender>
    <appender name="SIFT" class="ch.qos.logback.classic.sift.SiftingAppender">
        <!-- in the absence of the class attribute, it is assumed that the
             desired discriminator type is
             ch.qos.logback.classic.sift.MDCBasedDiscriminator -->
        <discriminator>
            <key>logid</key>
            <defaultValue>motorhead-core</defaultValue>
        </discriminator>

        <sift>
            <appender name="FILE-${logid}" class="ch.qos.logback.core.FileAppender">
                <file>${LOG_PATH}/${logid}.log</file>
                <append>false</append>
                <encoder>
                    <!--                    <pattern>%d [%thread] %level %mdc %logger{35} -%kvp -%msg%n</pattern>-->
                    <pattern>[%date{ISO8601}][%-24t][%-5level] %msg%n</pattern>
                </encoder>
            </appender>
        </sift>
    </appender>
    <appender name="LOKI" class="com.github.loki4j.logback.Loki4jAppender">
        <http>
            <url>http://localhost:3100/loki/api/v1/push</url>
        </http>
        <format>
            <label>
                <pattern>
                    cluster=%mdc{cluster:-none},job=%mdc{jobname:-none},jobnum=%mdc{jobnum:-none},host=${hostname},node=%mdc{node:-none},level=%level,thread=%t
                </pattern>
            </label>
            <message>
                <pattern>%msg %ex</pattern>
            </message>
        </format>
    </appender>
    <root level="info">
        <appender-ref ref="STDOUT"/>
        <appender-ref ref="SIFT"/>
        <appender-ref ref="LOKI"/>
    </root>
</configuration>

    Note: MDC values are used to label log messages, facilitating nodes and jobs correlation.

    Measuring and Monitoring

    LIBER*Batch exposes metrics via Micrometer. To enable metrics, configure Spring to expose management endpoints.

    YAML
    management:
  endpoints:
    web:
      exposure:
        include: "*"
  metrics:
    enable:
      jvm: true
      process: true
      system: true
      jdbc: true
      http: true
      logback: true
    tags:
      region: ${spring.application.name}
      environment: "test"
  tracing:
    enabled: false
    sampling:
      probability: 1.0

    Notes: - LIBER*Batch exposes additional custom metrics beyond standard Spring metrics. - Metrics are exposed through Spring Actuator endpoints—ensure proper security in production. - Enable tracing only when distributed tracing is required, as it impacts performance. - Batch application connectivity (i.e. JDBC for E/SQL commands) is configured through the application configuration file(s).
