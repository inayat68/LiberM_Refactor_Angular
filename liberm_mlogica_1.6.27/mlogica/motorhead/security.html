    Security

    LIBER*Batch manages security using Spring's standard authentication and authorization mechanisms. It supports four modes of operation:

    • disabled: no authentication or authorization is performed (default).
    • basic: basic HTTP authentication and RBAC authorization handled using an internal user store defined in a JSON file.
    • jwt-auth: authentication is performed using JSON Web Tokens (JWT), typically issued by an external identity provider. No RBAC authorization is applied.
    • jwt-rbac: authentication is performed using JWTs from an external identity provider, and RBAC authorization is enforced based on the roles present in the JWT.
    • custom: the LIBER*Batch web security component is not activated, for the user to provide their own Web Security component

    Disabled

    When security mode is set to disabled, no authentication or authorization is performed. This is the default mode of operation.

    Example configuration

    YAML
    motorhead:
  # ...
  security:
    mode: disabled

    Basic

    In basic mode, LIBER*Batch uses HTTP basic authentication and manages RBAC authorization using a user store.

    Users and their roles are specified in a JSON file, whose path is set with the motorhead.security.users configuration property.

    Each user in the JSON file is defined as a JSON object, where the key is the username and the value is an object containing:

    • password: the user's password, encoded with bcrypt and prefixed by {bcrypt}.
    • email: the user's email address.
    • roles: an array listing the roles assigned to the user.

    Example configuration

    YAML
    motorhead:
  # ...
  security:
    mode: basic
    users: 'file:./users.json'

    Example users file

    JSON
    {
  "admin": {
    "password": "{bcrypt}$2a$12$ogEqKNEJYP4xHwv7BWyHpeGuizx4.s7tf2VYnv4.hkHqLAu99HQmm",
    "email": "hand.of.doom@nib-labs.io",
    "roles": [
      "view",
      "submit",
      "alter"
    ]
  },
  "scheduler": {
    "password": "{bcrypt}$2a$12$fDBhxESlLC5.ebgRB0kQNuo0RQEtqIg1YrTlFl3T5X70JvL4GpwZO",
    "email": "herr.professor@nib-labs.io",
    "roles": [
      "submit"
    ]
  },
  "query": {
    "password": "{bcrypt}$2a$12$fDBhxESlLC5.ebgRB0kQNuo0RQEtqIg1YrTlFl3T5X70JvL4GpwZO",

    OAUTH2/JWT authentication only

    In jwt-auth security mode, LIBER*Batch authenticates users using JSON Web Tokens (JWT) issued by an external identity provider. In this mode, RBAC authorization is not applied.

    OAUTH2/JWT authentication is set up using standard Spring configuration properties. At a minimum, you must specify the JWK Set URI so that LIBER*Batch can validate JWT tokens.

    Example configuration

    YAML
    spring:
  # ...
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: http://localhost:9010/realms/demo

motorhead:
  # ...
  security:
    mode: jwt-auth

    OAUTH2/JWT with RBAC

    In jwt-rbac security mode, LIBER*Batch authenticates users using JSON Web Tokens (JWT) typically issued by an external identity provider. RBAC authorization is then enforced based on the roles included in the JWT.

    OAUTH2/JWT authentication is configured using standard Spring properties. At a minimum, you must specify the JWK Set URI so that LIBER*Batch can validate JWT tokens.

    Example configuration

    YAML
    spring:
  # ...
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: http://localhost:9010/realms/demo

motorhead:
  # ...
  security:
    mode: jwt-rbac
    jwt:
      client-id: testjes-client     # required if roles are namespaced per client

    Custom

    In custom security mode, LIBER*Batch uses a custom provided security component, that must define how users are authenticated and authorized.

    The custom provided component must be available on LIBER*Batch classpath and it package must be included in the component-scan. The component must provide a SecurityFilterChain type bean.

    Example configuration

    YAML
    motorhead:
  component-scan: org.sample.mhplugins
  # ...
  security:
    mode: custom
    Example security component

    Java
    @Configuration
@EnableWebSecurity
public class CustomWebSecurity {
    private static Logger log = LoggerFactory.getLogger(CustomWebSecurity.class);
    @Autowired
    private MotorheadConfig config;

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        log.debug("Custom web security active");
        return http
                .csrf(csrf -> csrf.disable())
                .authorizeHttpRequests((authorize) -> authorize
                        .anyRequest().permitAll()
                )
                .httpBasic(Customizer.withDefaults())
                .build();

    }
}

    Roles

    Roles are identified by a name and have associated permissions that determine access to commands and REST API endpoints.

    Their actual names can be defined in the configuration file via the motorhead.security.roles property.

    The available roles are:

    Role Property Default name Description Allowed APIs
    View view view Allows querying node/cluster status, classes, and job queues. Every APIs of type GET
    Alter alter alter Allows modifying node/cluster status, classes, and job queues. Every APIs of type PUT or POST
    Submit submit submit Allows submitting jobs. Only /job/submit

    The following example shows configuration for OAUTH2/JWT authentication with roles names definition:

    YAML
    spring:
  # ...
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: http://localhost:9010/realms/demo

motorhead:
  # ...
  security:
    mode: jwt-rbac
    roles:
      view: "batch-operator-view"
      alter: "batch-operator-alter"
      submit: "batch-operators-submit"

    The above configuration example customizes the role names used by LIBER*Batch to match those assigned by the identity provider. That means the users must have such roles assigned in the identity management software.

    Example setup with Keycloak

    This section provides a brief example of how to set up LIBER*Batch security using Keycloak as the identity provider.

    1. Install and run Keycloak (refer to the Keycloak documentation for instructions). You can use docker to run Keycloak quickly:

    Bash
    docker run -p 9010:8080 -e KEYCLOAK_ADMIN=admin -e KEYCLOAK_ADMIN_PASSWORD=admin quay.io/keycloak/keycloak:26.2 start-dev
    2. Access the Keycloak admin console at http://localhost:9010 and log in with the admin credentials. 3. Create a new realm (e.g., dev). 4. Create a new client (e.g., dev-client) with the following settings: - Client Protocol: openid-connect - Access Type: public - Valid Redirect URIs: * - Direct Access Grants Enabled: ON 5. Create roles corresponding to LIBERBatch roles (e.g., operator-view, operator-alter, operator-submit). 6. Create users. Make sure all necessary user info (email, name and surname) are provided and that the users have passwords set. For the example purposes, set "Email Verified" to "ON" for each user and make sure no "Required User Actions" are pending. 7. Assign the created roles to the users as needed. 8. Configure LIBERBatch to use the Keycloak instance you just configured for authentication and authorization by setting the following properties in the configuration file:

    YAML
      spring:
    # ...
    security:
      oauth2:
        resourceserver:
          jwt:
            issuer-uri: http://localhost:9010/realms/dev          # Check keycloak realm name here

  motorhead:
    security:
      mode: jwt-auth
      roles:
        view: "operator-view"
        alter: "operator-alter"
        submit: "operator-submit"
    1. Test the setup by obtaining a JWT token from Keycloak and using it to access LIBER*Batch APIs. You can obtain a token using the following curl command:
    Bash
       ACCESS_TOKEN=$(curl -s --location --request POST 'http://localhost:9010/realms/dev/protocol/openid-connect/token' \
        --header 'Content-Type: application/x-www-form-urlencoded' \
        --data-urlencode 'client_id=dev-client' \
        --data-urlencode 'username=john-doe' \
        --data-urlencode 'password=test' \
        --data-urlencode 'grant_type=password' |
        jq -r '.access_token')
  curl -X GET http://localhost:8081/cluster/status \
     -H "Authorization: Bearer $ACCESS_TOKEN"

    If you see the JSON response of the API, the setup is successful!
