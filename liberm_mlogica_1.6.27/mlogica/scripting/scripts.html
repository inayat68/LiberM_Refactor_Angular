    Jobs, steps, DDs, conditions, and restart

    LIBER*M job scripts are Groovy scripts based on NibJob. In practice, NibJob is the DSL entry point used to define a batch job and execute its steps.

    This page focuses on the executable script model. For the conceptual mapping from JCL to the Groovy DSL, see Script structure and JCL mapping.

    NibJob exposes the following core methods:

    JCL Groovy method
    JOB job
    EXEC exec
    DD dd

    The arguments intentionally use JCL-like names such as dsn, disp, cond, parm, and sysout.

    Declaration

    Job scripts are declared by annotating the script with @BaseScript NibJob.

    Example:

    Groovy
    import com.nib.groovybatch.NibJob
import groovy.transform.BaseScript

@BaseScript NibJob ACMEJOB

    Job

    The job method initializes the runtime job, loads the batch configuration, creates the job context, and optionally enables restart processing.

    Arguments:

    Argument Optional Description
    cond yes Job conditions, using JCL-like semantics
    userid yes User ID associated with the job
    restart yes Restart request, either explicit in the script or injected by the runtime context
    Groovy
    import com.nib.groovybatch.NibJob
import groovy.transform.BaseScript

@BaseScript NibJob ACMEJOB

job "COND02", cond: [8, LE]

    Steps

    Steps are declared through exec. A step can execute either:

    • a program through pgm:
    • a PROC through proc:

    Arguments:

    Argument Optional Description
    - no the name of the step (first argument)
    pgm yes the program to execute
    proc yes the PROC to execute
    cond yes step conditions
    parm yes step parameters
    - yes closure containing the DD statements

    Example:

    Groovy
    // //STEP05 EXEC PGM=IDCAMS
// //SYSPRINT DD   SYSOUT=*
// //SYSIN    DD   *
//    DELETE AWS.CCDA.CUSTDATA.CLUSTER -
//           CLUSTER                     
// /*

exec "STEP05", pgm: "IDCAMS", {
    dd "SYSPRINT", sysout: "*"
    dd "SYSIN", instream: [
            "   DELETE AWS.CCDA.CUSTDATA.CLUSTER -",
            "          CLUSTER                     "
    ]
}

    Step execution behavior

    At runtime, NibJob creates a Step object for every exec statement. The step tracks:

    • step name
    • step number
    • execution status (Ready, Running, Bypassed, Executed, Abend)
    • DD assignments
    • return code
    • timing information

    If the step invokes a PROC, the step is still part of the same runtime job model. External PROC scripts execute through NibProc, while instream PROCs execute as Groovy closures.

    Conditions

    Job and step conditions reproduce the semantics of JCL COND processing.

    Conditions can be expressed as:

    • a single operator
    • a list such as [8, LE]
    • a list including a target step such as [8, LE, "STEP010"]
    • multiple conditions

    Supported operators are:

    • GT
    • GE
    • EQ
    • LE
    • LT
    • NE
    • EVEN
    • ONLY

    Examples:

    Groovy
    job "CONDJOB", cond: [8, LE]

exec "STEP020", pgm: "PGM2", cond: [4, GT]

exec "STEP030", pgm: "PGM3", cond: [[8, LE, "STEP010"], [4, GT, "STEP020"]]

    Runtime behavior is consistent with the framework job model:

    • job conditions apply to the whole job unless a step condition overrides them
    • ONLY runs only after an abend is detected
    • EVEN allows execution after an abend
    • when restart is active, restart logic is evaluated before normal condition bypass rules

    DDs

    DDs are declared through the dd method. A DD can represent datasets, path-based files, SYSOUT, dummy DDs, instream content, closure-based processing, or concatenations.

    Arguments:

    Argument Optional Description
    - no the name of the DD (first argument)
    dsn yes the DSN name
    sysout yes the SYSOUT
    hold yes if true, SYSOUT is HOLD
    instream yes the instream content
    disp yes the dispositions
    dummy yes if true, file is considered a dummy
    gdg yes the GDG version
    lrecl yes the record length
    recfm yes the record format
    path yes the file path
    instream yes the instream content
    member yes the PDS member
    cached yes for VSAM files only, enables the file cache
    conversion yes file conversion at open (None, Ascii2Ebcdic, Ebcdic2Ascii)
    closure yes closure-based DD content, used for utilities such as NIBGRUVY

    Example:

    Groovy
    // //STEP01  EXEC PGM=IEBGENER
// //*
// //SYSUT1   DD *
// ADMIN001MARGARET            GOLD                PASSWORDA
// ADMIN002RUSSELL             RUSSELL             PASSWORDA
// ADMIN003RAYMOND             WHITMORE            PASSWORDA
// ADMIN004EMMANUEL            CASGRAIN            PASSWORDA
// ADMIN005GRANVILLE           LACHAPELLE          PASSWORDA
// USER0001LAWRENCE            THOMAS              PASSWORDU
// USER0002AJITH               KUMAR               PASSWORDU
// USER0003LAURITZ             ALME                PASSWORDU
// USER0004AVERARDO            MAZZI               PASSWORDU
// USER0005LEE                 TING                PASSWORDU
// /*
// //SYSUT2   DD DSN=AWS.M2.CARDDEMO.USRSEC.PS,
// //            DISP=(NEW,CATLG,DELETE),
// //            DCB=(LRECL=80,RECFM=FB,DSORG=PS,BLKSIZE=0),
// //            UNIT=SYSDA,SPACE=(TRK,(10,5),RLSE)
// //*
// //SYSPRINT DD SYSOUT=*
// //SYSIN    DD DUMMY

exec "STEP01", pgm: "IEBGENER", {
    // Instream content is expanded below
    dd "SYSUT1", instream: [
            "ADMIN001MARGARET            GOLD                PASSWORDA",
            "ADMIN002RUSSELL             RUSSELL             PASSWORDA",
            "ADMIN003RAYMOND             WHITMORE            PASSWORDA",
            "ADMIN004EMMANUEL            CASGRAIN            PASSWORDA",
            "ADMIN005GRANVILLE           LACHAPELLE          PASSWORDA",
            "USER0001LAWRENCE            THOMAS              PASSWORDU",
            "USER0002AJITH               KUMAR               PASSWORDU",
            "USER0003LAURITZ             ALME                PASSWORDU",
            "USER0004AVERARDO            MAZZI               PASSWORDU",
            "USER0005LEE                 TING                PASSWORDU"
    ]
    dd "SYSUT2", dsn: "AWS.M2.CARDDEMO.USRSEC.PS", disp: [NEW, CATLG, DELETE], lrecl: 80, recfm: FB
    dd "SYSPRINT", sysout: "*"
    dd "SYSIN", dummy: true
}

    DD naming and PROC overrides

    The framework supports both plain DD names and qualified names of the form STEP.DDNAME.

    Qualified names are especially important when overriding DDs inside a PROC invocation:

    Groovy
    dd "PRC001.FILEIN", dsn: "MY.INPUT.FILE", disp: SHR

    Internally, the runtime keeps both the DD name and, when provided, the target step name so that PROC step overrides can be resolved correctly.

    Concatenated DDs

    Concatenated DDs are defined by passing a list of DD descriptors:

    Groovy
    dd "SYSUT1", [
        [dsn: "APP.INPUT.PS", disp: SHR],
        [dsn: "APP.INPUT.PDS", member: "MBR1", disp: SHR]
]

    The runtime creates a primary DD and concatenates the remaining definitions to it.

    Special DD behaviors

    The framework supports several behaviors that are useful during migration:

    • temporary datasets when dsn starts with &&
    • GDG resolution through gdg:
    • instream content through instream:
    • spooled output through sysout: and optional output: descriptors
    • dummy DDs through dummy: true
    • path-backed files through path:
    • closure-backed DDs through closure:

    Output

    JCL OUTPUT statements are represented as maps passed to the dd method.

    If more than one OUTPUT statement are provided to the DD, use an array.

    See Spool for more details.

    Groovy
    // //STEP01  EXEC PGM=IEBGENER
// //SYSUT1   DD *
// ADMIN001MARGARET            GOLD                PASSWORDA
// ADMIN002RUSSELL             RUSSELL             PASSWORDA
// //OUT1   OUTPUT  CLASS=A,FORMS=GL000014
// //SYSUT2 DD  SYSOUT=(,),HOLD=YES,OUTPUT=(*.OUT1)                
// //SYSPRINT DD SYSOUT=*
// //SYSIN    DD DUMMY
exec "STEP01", pgm: "IEBGENER", {
    dd "SYSUT1", instream: [
            "ADMIN001MARGARET            GOLD                PASSWORDA",
            "ADMIN002RUSSELL             RUSSELL             PASSWORDA",
    ]
    def out1 = [CLASS: 'A', FORMS: 'XR000014']
    dd "SYSUT2", sysout: "(,)", hold: true, output: out1
    dd "SYSPRINT", sysout: "*"
    dd "SYSIN", dummy: true
}
// //STEP02  EXEC PGM=IEBGENER
// //SYSUT1   DD *
// ADMIN001MARGARET            GOLD                PASSWORDA
// ADMIN002RUSSELL             RUSSELL             PASSWORDA
// //OUT1   OUTPUT  CLASS=A,FORMS=GL000014
// //OUT2   OUTPUT  CLASS=A,FORMS=FR000014
// //SYSUT2 DD  SYSOUT=(,),HOLD=YES,OUTPUT=(*.OUT1,*.OUT2)                
// //SYSPRINT DD SYSOUT=*
// //SYSIN    DD DUMMY
exec "STEP01", pgm: "IEBGENER", {
    dd "SYSUT1", instream: [
            "ADMIN001MARGARET            GOLD                PASSWORDA",
            "ADMIN002RUSSELL             RUSSELL             PASSWORDA",
    ]
    def out1 = [CLASS: 'A', FORMS: 'XR000014']
    def out2 = [CLASS: 'B', FORMS: 'FRM00014']
    dd "SYSUT2", sysout: "(,)", hold: true, output: [out1, out2]
    dd "SYSPRINT", sysout: "*"
    dd "SYSIN", dummy: true
}

    Job restart

    The framework supports restart when scripting.restartEnabled is enabled in configuration.

    The restart request can come from:

    • the restart argument of job
    • the injected job context, for example when running under an external scheduler or orchestration layer

    The restart can be expressed as:

    • a string, indicating the step name to restart from. In this case, the job info is restored from the latest job executed which name matches the current job name.
    • a RestartData object, containing the following attributes:
      • step, the step name to restart from
      • jobid, the UUID of the job execution to restore
    • an integer, indicating the step number to restart from where to restore the job info.

    When restart is enabled, the runtime reloads a previously executed job, identifies the target step, and bypasses all preceding non-PROC steps until the restart point is reached. At restart time, the runtime also restores the tracked GDG allocation state from the reloaded job metadata.

    Examples:

    Groovy
    @BaseScript NibJob JRESTART

job "JRESTART", restart: "STEP03"   // restart from last execution
    Groovy
    @BaseScript NibJob JRESTART

job "JRESTART", restart: restart("STEP03", "d8eac78b-b2e0-4afb-a26e-3f8cd59d8be4")   // restart from a specific execution

    To restart a step defined in a PROC, the full name of the step (PROCNAME.STEPNAME) must be used. The job will restart from the first step which full name matches the requested one.

    When the job runs under LIBERBatch, the job restart can be triggered by the LIBERBatch API or command line, without editing the script.

    The job restart feature must be explicitly enabled in the scripting configuration.

    Example configuration:

    JSON
    {
  // nibbatch.json
  // ...
  "scripting": {
    "verbose": false,
    "restartEnabled": true,
    "localJobStore": "./tmp"
  }
}

    Utility methods and fields

    NibJob also exposes helper fields and methods that are useful in advanced scripts or when troubleshooting migrated jobs.

    Method/Field Description
    currentJob the current Job object
    currentProc the current Proc object, when executing inside a PROC
    currentStep the current Step object
    getJOBNAME() returns the job name
    getSTEPNAME() returns the current step name
    getSTEP(<name>) returns the Step object of the named step
    instream(<s>, <s>, ...) returns an instream object built with the provided arguments
    NOW() returns the current timestamp
    getOSName() returns the OS name (Linux, Windows, Unknown)
    RC() returns the job last executed step RC
    RC(<stepname>) returns the return code of the named step
    isABEND() returns whether an abend has been detected in the job
    isABEND(<stepname>) returns whether the named step abended
    ABENDCC() returns the maximum abend condition code, when available
    restart(<step>, <jobid>) returns the a RestartData object to use with restart option

    NibJob also publishes symbolic constants for common JCL concepts such as:

    • dispositions: SHR, NEW, OLD, KEEP, DELETE, CATLG, UNCATLG, MOD, PASS
    • condition operators: GT, GE, EQ, LE, LT, NE, EVEN, ONLY
    • organizations: PS, PO, VSAM
    • record formats: FB, FBA, VB, VBA, LSEQ
