    Script structure and JCL mapping

    This page explains how a traditional JCL job maps to the LIBER*M scripting DSL.

    The goal is not to reproduce JCL syntax literally, but to preserve the same execution structure and the same operational meaning.

    Mental model

    In LIBER*M, a batch job is typically represented as a Groovy script that extends NibJob.

    At a high level:

    • a JCL job becomes a Groovy job script
    • an EXEC PGM=... becomes exec ..., pgm: ...
    • an EXEC PROC=... becomes exec ..., proc: ...
    • a DD statement becomes dd ...
    • COND, SYSOUT, OUTPUT, instream data, and restart are expressed as Groovy arguments

    Structural mapping

    JCL concept LIBER*M scripting concept Notes
    JOB job "NAME", ... Initializes the runtime job
    EXEC PGM=... exec "STEP", pgm: "PGM", ... Executes a program or utility
    EXEC PROC=... exec "STEP", proc: PROCNAME, ... Executes an instream or external PROC
    DD dd "DDNAME", ... Defines file, SYSOUT, instream, or dummy DD
    COND= cond: ... Supported at job or step level
    PARM= parm: ... Passed to the executed program
    SYSOUT=* or SYSOUT=(,) sysout: "*" or sysout: "(,)" Supports spool integration
    OUTPUT output: map or output: [map1, map2] Used on SYSOUT DDs
    PROC DD override dd "STEP.DDNAME", ... Required for PROC step override
    RESTART= restart: ... Enabled through configuration

    Minimal job example

    JCL

    JCL
    //ACMEJOB  JOB (ACCT),'BATCH JOB',CLASS=A,MSGCLASS=X
//STEP01   EXEC PGM=IEBGENER
//SYSUT1   DD *
HELLO WORLD
/*
//SYSUT2   DD DSN=APP.OUTPUT.FILE,
//            DISP=(NEW,CATLG,DELETE),
//            DCB=(LRECL=80,RECFM=FB)
//SYSPRINT DD SYSOUT=*
//SYSIN    DD DUMMY

    LIBER*M script

    Groovy
    import com.nib.groovybatch.NibJob
import groovy.transform.BaseScript

@BaseScript NibJob ACMEJOB

job "ACMEJOB"

exec "STEP01", pgm: "IEBGENER", {
    dd "SYSUT1", instream: [
            "HELLO WORLD"
    ]
    dd "SYSUT2", dsn: "APP.OUTPUT.FILE", disp: [NEW, CATLG, DELETE], lrecl: 80, recfm: FB
    dd "SYSPRINT", sysout: "*"
    dd "SYSIN", dummy: true
}

    Script anatomy

    A typical job script has four parts:

    1. imports and base script declaration
    2. job declaration
    3. step definitions
    4. DD definitions inside each step

    Example:

    Groovy
    import com.nib.groovybatch.NibJob
import groovy.transform.BaseScript

@BaseScript NibJob MYJOB

job "MYJOB", cond: [8, LE]

exec "STEP010", pgm: "IDCAMS", {
    dd "SYSPRINT", sysout: "*"
    dd "SYSIN", instream: [
            "DELETE APP.TEST.CLUSTER CLUSTER"
    ]
}

    Mapping by section

    1. JOB

    The JCL JOB statement becomes a call to job.

    Example:

    Groovy
    job "ACMEJOB", userid: "BATCHUSR", cond: [8, LE]

    This initializes the runtime Job object and loads the framework configuration.

    2. EXEC PGM

    A JCL program step becomes an exec call with pgm:.

    Groovy
    exec "STEP010", pgm: "IDCAMS", parm: "OPTION=VALUE", {
    dd "SYSPRINT", sysout: "*"
}

    3. EXEC PROC

    A PROC invocation becomes an exec call with proc:.

    The proc value can be:

    • a Groovy closure for an instream PROC
    • a NibProc class/script for an external PROC
    Groovy
    exec "STEP020", proc: REPROC, CNTLLIB: "APP.CNTL", {
    dd "PRC001.FILEIN", dsn: "APP.INPUT.KSDS", disp: SHR
}

    4. DD

    Each DD is declared with dd.

    Common mappings:

    JCL pattern Groovy DSL
    DD DSN=FILE,DISP=SHR dd "DD1", dsn: "FILE", disp: SHR
    DD DISP=(NEW,CATLG,DELETE) disp: [NEW, CATLG, DELETE]
    DD SYSOUT=* dd "SYSPRINT", sysout: "*"
    DD DUMMY dd "SYSIN", dummy: true
    DD * dd "SYSIN", instream: [...]
    DD DSN=PDS(MEMBER) dd "SYSIN", dsn: "PDS", member: "MEMBER"

    PROC overrides

    One of the most important migration patterns is overriding DDs inside a PROC.

    In JCL this is done with qualified names such as STEP1.SYSIN. In LIBER*M, the same concept is expressed with a qualified DD name:

    Groovy
    dd "PRC001.SYSIN", dsn: "APP.CNTL", member: "JOBCTL", disp: SHR

    This is the standard way to override a DD belonging to a step inside a PROC.

    OUTPUT and spool mapping

    JCL OUTPUT statements are represented as maps attached to SYSOUT DDs.

    Groovy
    def out1 = [CLASS: 'A', FORMS: 'GL000014']

dd "SYSUT2", sysout: "(,)", hold: true, output: out1

    If more than one OUTPUT descriptor is needed, pass a list.

    See Spool for details.

    Conditions and bypass logic

    COND semantics are supported both at job level and at step level.

    Examples:

    Groovy
    job "JOB1", cond: [8, LE]
exec "STEP020", pgm: "PROG2", cond: [4, GT]
exec "STEP030", pgm: "PROG3", cond: ONLY

    The runtime evaluates the conditions against the job execution history and bypasses steps when required.

    Restart mapping

    JCL restart intent maps to the restart: argument of job.

    Examples:

    Groovy
    job "REJOB", restart: "STEP030"
    Groovy
    job "REJOB", restart: restart("STEP030", "d8eac78b-b2e0-4afb-a26e-3f8cd59d8be4")

    Restart must be enabled in configuration. When active, the runtime reloads the stored job metadata and resumes from the requested step.

    Utility steps vs application program steps

    From the script author perspective, both application programs and utilities are invoked with exec ... pgm: ....

    Examples:

    Groovy
    exec "SORT01", pgm: "NIBSORT", {
    dd "SORTIN", dsn: "APP.INPUT", disp: SHR
    dd "SORTOUT", dsn: "APP.OUTPUT", disp: [NEW, CATLG, DELETE], lrecl: 80, recfm: FB
    dd "SYSIN", instream: [
            "  SORT FIELDS=(1,10,CH,A)"
    ]
}
    Groovy
    exec "LOAD01", pgm: "MYCOBOLP", parm: "RUNMODE=FULL", {
    dd "INPUT", dsn: "APP.INPUT.FILE", disp: SHR
}

    The difference is in the resolved program implementation, not in the script syntax.
