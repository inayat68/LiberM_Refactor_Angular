    Region & Node

    LIBERTP is configured using standard Spring Boot configuration properties and a set of collateral configuration files that define the LIBERTP resources.

    Application Properties

    The master configuration is contained in the Spring application properties, and refers to the resource definitions in the collateral configuration files.

    The application configuration follows the general Spring Boot configuration structure and rules. Therefore, multiple configuration files can be used to define different profiles and configurations.

    The configuration properties are prefixed with supernaut and are used to configure the LIBER*TP subsystems and components.

    The configuration must include the following sets of properties:

    1. Spring properties to configure the Spring Boot application.
    2. Server properties to configure the server.
    3. Management properties to configure the management endpoints.
    4. LIBER*TP properties to configure the LIBER*TP subsystems and components.

    LIBER*TP Application Properties

    supernaut.backbone.storage The backbone storage type (embedded, redis, hazelcast). none
    supernaut.backbone.config The backbone configuration string. none
    supernaut.backbone.shardingStrategy The key sharding strategy. none
    supernaut.backbone.redisConfigProvider The class name of the a Redis configuration provider. none
    supernaut.programs.packages The packages to scan for LIBER*TP programs. none
    supernaut.programs.classpath The classpath to scan for LIBER*TP programs. none
    supernaut.programs.dependencies The classpath to scan for LIBER*TP program dependencies. none
    supernaut.component-scan The packages to scan for Spring components. none
    supernaut.transactions The path to the transactions configuration file (PCT). none
    supernaut.files The path to the files configuration file (FCT). none
    supernaut.tdqueues The path to the TD queues configuration file (DCT). none
    supernaut.plt The path to the program loading table configuration file. none

    supernaut.terminals.typeterm The path to the terminal types configuration file. none

    supernaut.terminals.codepage The codepage for the EBCDIC/ASCII conversion embedded ENUS037

    Example Configuration

    YAML
    spring:
  application:
    name: TESTCICS
  profiles:
    active: dev
  cloud:
    config:
      enabled: false
    function:
      scan:
        packages: "org.sample.supernaut,com.nib.supernaut"
  lifecycle:
    timeout-per-shutdown-phase: "10s"

server:
  shutdown: graceful

management:
  endpoints:
    web:
      exposure:
        include: "*"
  metrics:
    enable:
      tomcat: false
      jvm: true
      process: true
      hikaricp: false
      system: true
      jdbc: true
      http: true
      logback: true
    tags:
      region: ${spring.application.name}
      environment: "test"
  #    distribution:
  #      percentiles-histogram:
  #        http:
  #          server:
  #            requests=true:
  tracing:
    enabled: true
    sampling:
      probability=: 1.0

supernaut:
  applid: ${spring.application.name}
  sysid: "DARK"
  cwa: 512
  programs:
    packages: >
      org.sample.online.testapp,
      org.sample.online.testapp2
    classpath: >
      ${NIB_JAVA_HOME}/samples/sample-online/target/classes,
      ${NIB_JAVA_HOME}/nib-commons-cobol/target/classes,
    dependencies: >
      ${NIB_JAVA_HOME}/nib-commons-cobol/lib
  component-scan: >
    org.sample.online.servlets,
    org.sample.online.webservices
  transactions: 'file:./pct.json'
  files: 'file:./fct.json'
  tdqueues: 'file:./dct.json'
  terminals:
    tct: 'file:./tct.json'
    typeterm: 'file:./typeterm.json'
    timeout: 1
    # codepage: "classpath:/codepages/ENUS037.cpg"
  bms:
    path: "classpath:/mapsets"
    attributes:
      ascii: false
      convert_space: false
  doctemplate:
    config: 'file:./doctemplate.json'
    path: "classpath:/doctemplates"
  extensions:
    iscobol:
      enabled: false
      cobpath: "${NIB_JAVA_HOME}/samples/sample-iscobol/target/classes"
  security:
    users:
      default: "CICSUSER"
      system: "SYSTEM"
    authorization:
      provider: "com.nib.supernaut.security.RoleBasedAuthorizationProvider"
      config: "file:./roles.json"
    authentication:
      provider: "com.nib.supernaut.security.RdbmsAuthenticationProvider"
      config: "jdbc:postgresql://localhost/postgres?user=postgres&password=postgres&ssl=false&currentSchema=online_security"
  spool:
    provider: "com.nib.supernaut.spool.DummySpoolProvider"
    config: "jdbc:postgresql://localhost/postgres?user=postgres&password=postgres&ssl=false&currentSchema=online_security"
  jms:
    ibm-mq: false
    trigger-monitors: 'file:./mq-trigger-monitors.json'

logging:
  config: file:./logback.xml
  level:
    com.nib: DEBUG

ibm:
  mq:
    queueManager: "QM1"
    channel: "DEV.ADMIN.SVRCONN"
    connName: "localhost(1414)"
    user: "admin"
    password: "passw0rd"

    Redis backbone client

    The Redis backbone uses the Redisson client to connect and use Redis. Of course such file must be defined according to the Redis configuration. For further information on Redisson, please refer to the Redisson documentation.

    For example, the following configuration matches a simple Redis setup with only one node:

    YAML
    singleServerConfig:
address: "redis://127.0.0.1:6379"

    This is another example with a redundant Redis configuration with replicas and three sentinels:

    YAML
    sentinelServersConfig:
  masterName: "mymaster"
  sentinelAddresses:
    - "redis://sentinel-1:26379"
    - "redis://sentinel-2:26379"
    - "redis://sentinel-3:26379"
  timeout: 3000
  masterConnectionPoolSize: 64
  slaveConnectionPoolSize: 64
