    Thread Pools and Database Connection Pool Sizing in {supernaut.name}

    Overview

    {supernaut.name} executes workloads originating from two different sources:

    1. Terminal-driven tasks (REST driven workloads, i.e. TN3270, Angular clients or REST CTG)
    2. Background tasks (asynchronous workloads, i.e. ATI, TCPIP services or MQ originated task)

    These workloads are executed on different thread pools, but they share the same database connection pool ( Atomikos). Correct sizing of these pools is essential to avoid contention, latency spikes, and thread pile-ups.

    This section describes how the main configuration parameters interact and provides recommended sizing guidelines.

    Execution Model

    REST originated tasks

    REST requests (for example TN3270 terminal interactions) are handled by the HTTP container thread pool.

    In a standard Spring Boot deployment, this is the Tomcat thread pool.

    Typical examples include:

    • TN3270 requests
    • Angular client requests
    • REST CTG requests

            Terminal request
               │
               ▼
        Tomcat Thread Pool
    (server.tomcat.threads.max)
               │
               ▼

       ▼                ▼
     E/SQL             VSAM 
    database         database
     access           access

    Background Tasks

    {supernaut.name} executes background workloads using the Spring Task Executor.

    Typical examples include:

    • MQ processing
    • CTG requests
    • internal scheduled tasks
    • asynchronous processing

               │
               ▼
   Spring ThreadPoolTaskExecutor
(spring.task.execution.pool.max-size)
               │

  {supernaut.name} task execution
       │                │
       ▼                ▼
     E/SQL             VSAM 
    database         database
     access           access

    Each running background task occupies one executor thread.

    Database Connection Pool

    All workloads share the same Atomikos connection pool.

    Text Only
    Tomcat threads ─┐
                │
Spring executor ├──► Atomikos connection pool
                │
Other tasks  ───┘

    Connection Pool Size

    This parameter maxPoolSize defines the maximum number of concurrent database connections.

    If the pool is exhausted:

    • threads block waiting for a connection
    • request latency increases
    • thread pools can become saturated

    Connection Wait Timeout

    The parameter borrowConnectionTimeout defines how long a thread waits for a connection.

    This value defines the maximum time (in seconds) a thread waits for a database connection from the pool.

    Behavior:

    • If a connection becomes available within the timeout, the task continues normally.
    • If the timeout expires, Atomikos throws a connection acquisition exception and the transaction fails.

    If the timeout is too large:

    • Tomcat threads may remain blocked for long periods
    • thread pools can accumulate waiting requests
    • latency can increase dramatically under load

    If the timeout is too small:

    • transient spikes may cause unnecessary transaction failures

    Recommended values

    For interactive workloads typical of {supernaut.name}:

    workload type recommended value
    online transactions 5–10 seconds
    batch / background workloads 10–30 seconds

    This ensures that connection pool saturation is detected quickly and prevents long thread blockages.

    Concurrency Model

    The maximum number of concurrent database operations is limited by the smallest of the following:

    Text Only
    min(
    Tomcat maxThreads,
    Spring executor maxPoolSize,
    Atomikos maxPoolSize
)

    However, in practice the limiting factor is usually Atomikos maxPoolSize, because it controls how many database transactions can execute simultaneously.

    Recommended Sizing Strategy

    Step 1 — Determine Online Concurrency

    Configure the Tomcat thread pool based on expected terminal activity.

    Example:

    YAML
    server:
  tomcat:
    threads:
      max: 100

    This allows up to 100 concurrent terminal requests.

    Step 2 — Configure Background Parallelism

    Limit background parallelism so it does not overwhelm the database.

    Example:

    YAML
    spring:
  task:
    execution:
      pool:
        core-size: 5
        max-size: 10
        queue-capacity: 0

    Typical values:

    workload type recommended max
    light background activity 5–10
    heavy integration workloads 10–20

    Step 3 — Size the Datasource Database Pool

    The Atomikos pool must support both terminal and background workloads.

    A good starting rule is:

    Text Only
    Atomikos maxPoolSize ≈ 25–40% of Tomcat maxThreads

    Example:

    Text Only
    Tomcat threads: 100
Background threads: 10
Atomikos maxPoolSize: 30–40

    This prevents excessive concurrency on the database while maintaining good throughput.

    Note: The Atomikos pool size is defined in Supernaut properties for each datasource pool. See Datasources.

    Example Configuration

    Example production configuration:

    YAML
    server:
  tomcat:
    threads:
      max: 100

spring:
  task:
    execution:
      pool:
        core-size: 5
        max-size: 10
        queue-capacity: 0
        keep-alive: 30s

    Connection pool configuration:

    YAML
    supernaut:
  datasources:
    connection-pools:
      min-pool-size: 5
      max-pool-size: 30
      borrow-connection-timeout: 10

    Concurrency limits:

    component limit
    terminal tasks 100
    background tasks 10
    DB transactions 30

    Avoiding Thread Pile-Ups

    If the Atomikos pool is too small relative to incoming requests:

    • Tomcat threads accumulate waiting for connections
    • latency increases sharply
    • the system may appear stalled

    The borrowConnectionTimeout parameter helps prevent long blocking periods by forcing requests to fail quickly when the connection pool is exhausted.

    If the pool is too large:

    • database contention increases
    • lock conflicts rise
    • query performance degrades

    A moderate connection pool is usually optimal.

    Monitoring Recommendations

    Monitor the following metrics:

    metric meaning
    Tomcat busy threads online concurrency
    TaskExecutor active threads background workload
    Atomikos active connections database load

    When tuning the system:

    • if Atomikos active ≈ maxPoolSize frequently → increase pool size
    • if DB CPU or lock contention rises → reduce pool size
    • if many requests fail with connection acquisition timeout → increase pool size or reduce concurrency

    Typical Production Values

    For many {supernaut.name} deployments:

    parameter typical value
    Tomcat maxThreads 80–150
    TaskExecutor maxPoolSize 5–15
    Atomikos maxPoolSize 25–40
    borrowConnectionTimeout 5–10 s

    Actual optimal values depend on:

    • database capacity
    • transaction duration
    • workload characteristics.
