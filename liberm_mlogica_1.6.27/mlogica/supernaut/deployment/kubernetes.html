    Deploying on Kubernetes

    LIBER*TP can be obviously deplyed on Kubernetes. The deployment requires the definition of:

    • A Deployment for the LIBER*TP nodes
    • A Service for the LIBER*TP nodes, to expose the node ports and allow the nodes to expose the API. Either a Loadbalancer or a ClusterIP.
    • An Ingress to route requests to your service (in case you are using a ClusterIP).

    Optionally, if you plan to use a Tn3270 gateway, you can also deploy a Deployment and a service for the Tn3270 gateway.

    Backbone

    As described in other documents, a LIBER*TP region requires a backbone.

    A backbone can be provided either as a pod of the application or an external service. So you can either deploy a " self-contained" k8s application, that includes the backbone and the LIBER*TP nodes, or you can have your backbone provided by, i.e., as an external SaaS such as AWS ElastiCache or a Redis database running on bare metal.

    Node Deployment

    As mentioned before, the LIBER*TP nodes are deployed as Deployment. We suggest to define a replica set sufficiently large to handle the workload of your application.

    YAML
    apiVersion: apps/v1
kind: Deployment
metadata:
  name: supernaut
  namespace: supernaut-demo
spec:
  replicas: 3
  selector:
    matchLabels:
      app: supernaut
  template:
    metadata:
      labels:
        app: supernaut
    spec:
      initContainers:
        - name: wait-for-redis
          image: busybox
          command: [ 'sh', '-c', 'until nc -z redis 6379; do echo waiting for redis; sleep 2; done' ]
      containers:
        - name: supernaut
          image: nib/supernaut
          imagePullPolicy: IfNotPresent
          ports:
            - containerPort: 8080
          env:
            - name: NIB_JAVA_LICENSE_KEY
              valueFrom:
                secretKeyRef:
                  name: nib-java-license
                  key: NIB_JAVA_LICENSE_KEY
          volumeMounts:
            - name: config-volume
              mountPath: /config
            - name: bms-volume
              mountPath: /bms
            - name: classes-volume
              mountPath: /classes
            - name: nib-volume
              mountPath: /opt/nib
          readinessProbe:
            httpGet:
              path: /actuator/health
              port: 8080
            initialDelaySeconds: 10
            periodSeconds: 5
            failureThreshold: 3
            timeoutSeconds: 2
      volumes:
        - name: config-volume
          configMap:
            name: supernaut-config
        - name: bms-volume
          hostPath:
            path: /run/desktop/mnt/host/d/nib/nib-java/samples/aws-carddemo-java/target/classes/bms
        - name: classes-volume
          hostPath:
            path: /run/desktop/mnt/host/d/nib/nib-java/samples/aws-carddemo-java/target/classes
        - name: nib-volume
          hostPath:
            path: /run/desktop/mnt/host/d/nib/nib-java

    As in the example, you can define a readiness probe to check if the node is up and running. The probe can check the /actuator/health endpoint, which is exposed by the Spring Boot actuator.

    Node configuration

    Node configuration can be done using the ConfigMap object. The ConfigMap is mounted in the pod as a volume, and the configuration is read from there.

    You can include any the configuration files in the ConfigMap

    • Node configuration (application.yml)
    • Files referenced by the configuration (i.e. the redis config, logging config, etc.)
    • Resources referenced by the configuration (i.e. pct.json, fct.json, etc.)
    • Any other application configuration files
    Text Only
    kubectl create configmap supernaut-config --namespace supernaut-demo --from-file=./supernaut

    Secrets

    Of course, Secrets are used to store sensitive information, such as the license key, database credentials, etc.

    For example, I could have my sensitive info stored in a Secret:

    Text Only
    kubectl create secret generic my-secret --namespace my-namespace --from-literal=DB_PASSWORD="MySecretPassword"

    Mount it in the pod as an environment variable:

    YAML
              env:
            - name: DB_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: my-secret
                  key: DB_PASSWORD

    And then use it in the supernaut configurations:

    JSON
    {
  "name": "pgsql-non-xa",
  "xa": false,
  "driver": "org.postgresql.Driver",
  "url": "jdbc:postgresql://host.docker.internal/postgres?user=postgres&password=${env:DB_PASSWORD}&ssl=false&currentSchema=darkside"
}

    Mount points

    The mount points are used to store the application classes, bms json files, the data files and the logs. Please make sure to mount all the required directories that will be used by the LIBER*TP nodes as well as by the application programs.

    Logging

    LIBER*TP logs via slf4j and is deployed by default with logback. Therefore, you have maximum flexibility in configuring how and where to log.

    In fact, you may configure logs to:

    • ship logs to a log aggregator (i.e. ELK, Splunk, Loki, etc.)
    • ship logs to a managed log service (i.e. AWS CloudWatch Logs, GCP Cloud Logging, etc.)
    • write logs just on files or stdout and gather via kubernetes or other log capture daemons

    For example, in this logback.xml file, we configure the application to log on files, on the mount point /logs:

    XML
    <configuration>
    <define name="hostname"
            class="ch.qos.logback.core.property.CanonicalHostNamePropertyDefiner"/>
    <appender name="STDOUT"
              class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <pattern>[%date{ISO8601}][%-24t][%-5level] %msg%n</pattern>
        </encoder>
    </appender>
    <logger name="com.nib" level="debug"/>
    <root level="info">
        <appender-ref ref="STDOUT"/>
    </root>
</configuration>

    Service

    The Service is used to expose the node ports and allow the nodes to expose the API. You can use either a LoadBalancer or a ClusterIP.

    YAML
    apiVersion: v1
kind: Service
metadata:
  name: supernaut
  namespace: supernaut-demo
  labels:
    app: supernaut
spec:
  selector:
    app: supernaut
  ports:
    - protocol: TCP
      port: 8880
      targetPort: 8080
  type: ClusterIP

    Ingress

    If you are using a ClusterIP service, then you need an Ingress to route HTTP requests from the external world to the cluster IP.

    YAML
    apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: supernaut-ingress
  namespace: supernaut-demo
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  rules:
    - host: supernaut.127.0.0.1.nip.io
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: supernaut
                port:
                  number: 8880

    This will handle all the request going to http://supernaut.127.0.0.1.nip.io/.

    Tn3270 Gateway

    If you plan to use a Tn3270 gateway, you can also deploy a Deployment and a Service for the Tn3270 gateway.

    The only mount point required is the config mount point, which is used to store the configuration files ( tn3270.yaml).

    YAML
    apiVersion: apps/v1
kind: Deployment
metadata:
  name: tn3270-gw
  namespace: supernaut-demo
spec:
  replicas: 1
  selector:
    matchLabels:
      app: tn3270-gw
  template:
    metadata:
      labels:
        app: tn3270-gw
    spec:
      containers:
        - name: tn3270-gw
          image: nib/tn3270
          imagePullPolicy: IfNotPresent
          ports:
            - containerPort: 3270
          env:
            - name: NIB_JAVA_LICENSE_KEY
              valueFrom:
                secretKeyRef:
                  name: nib-java-license
                  key: NIB_JAVA_LICENSE_KEY
          volumeMounts:
            - name: config-volume
              mountPath: /config
      volumes:
        - name: config-volume
          configMap:
            name: supernaut-config

    The Service must expose the TCP port used by tn3270 terminals to connect. Therefore, the service must be of type NodePort or LoadBalancer.

    YAML
    apiVersion: v1
kind: Service
metadata:
  name: tn3270-gw
  namespace: supernaut-demo
spec:
  type: LoadBalancer
  selector:
    app: tn3270-gw
  ports:
    - name: tn3270-gw
      protocol: TCP
      port: 3270
      targetPort: 3270
