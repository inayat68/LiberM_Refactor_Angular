    MQ Adapter

    MQ access is facilitated through the Spring/JMS API. While this approach offers limited flexibility compared to the native MQ API, it provides significant future benefits:

    • It is recommended by IBM as the preferred method for accessing MQ from Java.
    • It enables support for alternative messaging protocols (e.g., ActiveMQ).

    Access to MQ is ensured by the JMS Connection Factory provided by IBM's mq-jms-spring-boot-starter package.

    CICS MQ Adapter

    The MQ Adapter is implemented using a JMS Message Listener, which listens for MQ messages on the trigger queue. Upon receiving a message, it initiates the specified transaction, allowing access to the MQ queue using Spring JMS APIs (via JmsTemplate).

    The message posted on the trigger queue must be in MQTM struct V1 format.

    Configuration

    To configure MQ/Adapters, the folollowing settings must be updated.

    1. LIBER*TP Configuration properties
    2. MQ Queue Manager properties
    3. Trigger Monitors configuration

    LIBER*TP configuration

    To enable MQ, the region configuration must include the setting supernaut.jms.ibm-mq=true. Once MQ is activated, the region will initialize the MQ AutoConfiguration service, which establishes XA connections to the queue manager specified in the MQ properties (outlined below).

    Enabling MQ allows user programs to access the JMS APIs and communicate with MQ.

    YAML
    supernaut:
  jms:
    ibm-mq: true
    trigger-monitors: 'file:./mq-trigger-monitors.json'

    MQ properties

    MQ properties define the connection details for the queue manager and are specified in accordance with the IBM MQ JMS Spring Components.

    YAML
    ibm:
  mq:
    queueManager: "QM1"
    channel: "DEV.ADMIN.SVRCONN"
    connName: "localhost(1414)"
    user: "admin"
    password: "passw0rd"

    Trigger Monitors

    Trigger monitors are configured in a JSON file, which is specified in the LIBER*TP configuration. The file contains an array of trigger monitor configurations, each of which specifies the queue name and the program to be executed.

    JSON
    {
  "TriggerMonitors": [
    {
      "name": "TM01",
      "queue": "TEST.QUEUE.TRG.01",
      "userid": "ADMIN"
    },
    {
      "name": "TM02",
      "queue": "DEV.QUEUE.2",
      "userid": "ADMIN"
    }
  ]
}

    To configure trigger monitors, see MQ Trigger Monitors.

    Example customer program

    Transaction using MQI (including those started by the MQ Trigger Monitors) can access MQ using directly the JMS API.

    NOTE: of course MQI equivalent APIs can be created on top of JMS, but as long as we don't know how big MMF should be, let's assume a deeper transformation

    Example

    Java
    @Override
class WorkingStorage extends NGroup {
    public StructMqtm mqtm = new StructMqtm();
    final NInt16 intoLen = new NInt16();
    public NChar message = new NChar(40).initial(NChar.Space);
}

@ProgramStorage
final WorkingStorage WS = new WorkingStorage();

protected int procedure(AbstractNField dfhcommarea) throws ContextException {
    WS.intoLen.setValue(WS.mqtm.length());
    supernaut.retrieve()
            .into(WS.mqtm)
            .length(WS.intoLen)
            .nohandle()
            .exec();

    var mq = supernaut.jms();
    var hconn = new JmsConnection();
    JmsObjectDescriptor od = new JmsObjectDescriptor();
    od.setQueue(WS.mqtm.getQName().toString().trim());
    JmsObject hobj = null;
    try {
        hobj = mq.open(hconn, od, IJmsMqApi.OpenMode.InputShared);
        int l = mq.get(hconn, hobj, new MessageDescriptor(), new JmsGetOptions(), WS.message, WS.message.length());
        supernaut.trace("received %d bytes [%s]", l, WS.message.getString(0, l));
        mq.close(hconn, hobj, 0);
    } catch (JmsMqApiException e) {
        throw new RuntimeException(e);
    } catch (JMSException e) {
        throw new RuntimeException(e);
    }
    return 0;
}
